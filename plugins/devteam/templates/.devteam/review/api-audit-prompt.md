以下は仕様サマリと、実装コード・テストコードです。
あなたは第三者監査として、テストが品質保証として機能しているか検証してください。

目的:
- 「テストが通っているのに本番で壊れる」状態を防ぐこと

ルール:
- 推測でOKと言わない
- OKと言うなら必ず根拠を示す
- 根拠が出せないものは未網羅扱いにする
- 根拠は必ず「テスト名 +（該当ファイル名 / 関数名 / 該当箇所の短い引用）」で示す
- 提示されていないコード/テストに基づく判断は禁止。不明は未網羅として列挙する
- 監査は提示された「仕様サマリ」に書かれている PUBLIC API / RULES / BRANCHES のみを対象に行う

出力:
1. 仕様→テスト対応表
2. C1（分岐）対応表 + 未網羅
3. 型安全性の懸念
4. 冗長/危険
5. 仕様自体の欠陥
6. 追加すべき最小テスト
7. すり抜けリスク
