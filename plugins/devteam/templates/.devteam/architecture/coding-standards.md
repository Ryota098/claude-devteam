# Coding Standards

## 目的
このプロジェクトで、保守しやすく一貫性のある実装を行うための基準。

## ディレクトリ構成
- 各層の責務を明確に分ける
- UI、API、ドメイン、インフラを混在させない
- 共通処理は重複させず、責務の明確な場所へ切り出す

## 命名規則
- ファイル名、関数名、変数名は一貫した命名規則に従う
- 略語を乱用しない
- 意味が分かる名前を優先する

## 共通化ルール
- 2回以上同じ意図のコードが出たら共通化を検討する
- ただし早すぎる抽象化は避ける
- 共通化は利用側の責務を曖昧にしない範囲で行う

## 実装ルール
- 既存の設計方針に合わせる
- 新しいパターンを導入する場合は理由を明記する
- private な実装詳細を public API に漏らさない
- エラー処理、境界値、未定義入力を明示的に扱う

## テストルール
- 外部挙動を中心に検証する
- 実装詳細依存のテストを増やしすぎない
- 主要分岐ごとの検証を意識する

## 禁止事項
- 既存構造を無視した場当たり的な追加
- 命名ルールを崩す実装
- 目的が曖昧な共通化
- 根拠のない設計変更
