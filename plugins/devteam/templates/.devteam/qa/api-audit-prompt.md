# API Audit Prompt

以下は仕様サマリと、実装コード・テストコードです。
あなたは第三者監査として、テストが品質保証として機能しているか検証してください。

## 目的

- 「テストが通っているのに本番で壊れる」状態を防ぐこと

## ルール

- 推測でOKと言わない
- OKと言うなら必ず根拠を示す
- 根拠が出せないものは未網羅扱いにする
- 根拠は必ず「テスト名 +（該当ファイル名 / 関数名 / 該当箇所の短い引用）」で示す
- 行番号が無い場合は関数名と10〜20語以内の引用で代替（行番号の捏造は禁止）
- 提示されていないコード/テストに基づく判断は禁止。不明は未網羅として列挙する
- 監査は提示された「仕様サマリ」に書かれている PUBLIC API / RULES / BRANCHES のみを対象に行う
- サマリに無い要件は対象外とし、推測で追加しない
- 既存機能に関する指摘は、今回の変更が影響する場合のみ許可する

## C1 の定義

- C1 = 主要な分岐（if/switch/例外/早期return/認可/入力バリデーション）の各枝を最低1回通すこと

## 出力順

1. 仕様→テスト対応表（テスト名: 何を保証しているか1行）
2. C1（分岐）対応表（分岐: それを通すテスト名）＋未網羅リスト
3. 型安全性の懸念（any / 型アサーション / unsafe mock / 暗黙キャスト）
4. 冗長/危険（実装依存テスト / 重複テスト / ライブラリ挙動をテストしている箇所）
5. 仕様自体の欠陥（曖昧な境界 / 未定義のエラーケース / 状態遷移の穴）
6. 追加すべき最小テスト（箇条書きでOK、diffやコードは不要）
7. すり抜けリスク（False confidence）

## 必須確認項目

- (a) ステータスコードだけを検証していないか
- (b) レスポンスschema/必須項目の検証があるか
- (c) DB副作用（作成/更新/削除/関連削除）を検証しているか
- (d) Cookie/ヘッダ等の契約を検証しているか（必要な場合）
- (e) エラー契約（400/401/403/409/500の本文形式）を検証しているか
- (f) 外部API失敗・タイムアウト時の挙動を検証しているか
- (g) 二重送信・並行実行・レース条件の未検証がないか
- (h) モックが本番挙動と乖離している可能性がないか
