# Frontend Audit Prompt

以下は仕様サマリと実装コードです。テストコードはありません。
あなたは第三者監査として、実装が仕様を正しく満たしているか、
「動いているように見えて本番で壊れる」状態を検出してください。

## ルール

- 推測でOKと言わない。OKなら必ず根拠（コンポーネント名/関数名/10〜20語の引用）を示す
- 根拠が出せないものは未網羅扱い
- 提示されていないコード/ファイルに基づく判断は禁止。不明は未網羅として列挙
- 監査は仕様サマリの PUBLIC API / RULES / BRANCHES のみを対象にする
- サマリにない要件は対象外。推測で追加しない

## C1 の定義

- 主要な分岐（if/switch/三項演算子/早期return/条件付きレンダリング/エラーハンドリング）の各枝を最低1回通すこと

## 出力順

1. 仕様→実装対応表
   - コンポーネント/関数名: 何を実現しているか1行
2. C1（分岐）網羅表
   - 分岐 | 実装上の確認可否 | 確認できない理由
3. 状態管理の懸念
4. 非同期・ライフサイクルの懸念
5. コンポーネント間連携の懸念
6. 型安全性の懸念
7. 仕様自体の欠陥
8. 追加すべき最小検証項目（テスト観点・箇条書き）
9. すり抜けリスク（False confidence）

## 必須確認項目

- (a) loading/error/空/正常の4状態を全てハンドリングしているか
- (b) フォーム送信の二重送信防止が実装されているか
- (c) アンマウント後のstate更新が考慮されているか
- (d) APIレスポンスのschema検証があるか（型アサーションで誤魔化していないか）
- (e) エラーメッセージがユーザーに適切に伝わるか（500系の扱いを含む）
- (f) 外部API失敗・タイムアウト時のUI挙動が定義されているか
- (g) 認証トークン期限切れ・未認証時のリダイレクト/表示が考慮されているか
- (h) 並行実行・race条件（古いレスポンスが新しいものを上書きしないか）
- (i) global storeへの変更が意図しない他コンポーネントに波及していないか
- (j) useEffectのdependency配列が実際の依存と一致しているか
