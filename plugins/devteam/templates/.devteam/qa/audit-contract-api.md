# API Audit Contract

## Goal
- テストが通っていても本番で壊れる状態を防ぐ

## Rules
- 推測で OK と言わない
- OK と言うなら根拠を示す
- 根拠が出せないものは未網羅扱いにする
- 根拠はテスト名 + ファイル名 / 関数名 / 該当箇所の短い引用で示す
- 提示されていないコードやテストに基づく判断は禁止
- 監査対象は仕様サマリに書かれている PUBLIC API / RULES / BRANCHES のみ

## C1 Definition
- 主要な分岐（if/switch/例外/早期return/認可/入力バリデーション）の各枝を最低1回通すこと

## Audit Output Order
1. 仕様→テスト対応表
2. C1（分岐）対応表 + 未網羅リスト
3. 型安全性の懸念
4. 冗長/危険
5. 仕様自体の欠陥
6. 追加すべき最小テスト
7. すり抜けリスク

## Required Risk Checks
- ステータスコードだけを検証していないか
- レスポンス schema / 必須項目の検証があるか
- DB副作用を検証しているか
- Cookie / header 等の契約を検証しているか
- エラー契約を検証しているか
- 外部API失敗・タイムアウト時の挙動を検証しているか
- 二重送信・並行実行・race 条件の未検証がないか
- モックが本番挙動と乖離していないか
